# Paper 0.75 - Applying Tools as Next-State Preconditions

## Purpose

Paper 0.75 explains how a result from one review interval becomes a
precondition for the next. The goal is to make transitions visible: shell
changes, escrow, delay, boundary collision, recentering, and higher-tier
framing should be named rather than implied.

## Next-State Rule

When a paper produces a result, the next paper may use it only as one of the
following:

```text
proved precondition
bounded local result
boundary receipt
escrowed obligation
delayed dependency
candidate transform
recentered state
```

This prevents an informal observation from becoming a hidden premise.

## Preconditioned Settings

A preconditioned setting is a review state that has already been shaped by a
previous paper. Examples include:

- a known carrier,
- a validated correction surface,
- a closed finite inventory,
- a boundary receipt,
- a kernel output,
- a lib binding,
- an analog exposure sheet,
- an unresolved obligation.

The next paper may start from that setting only if it names the previous
receipt or source binding.

## Shell, Escrow, and Delay

Some results are not failures and not proofs. They are delayed states:

```text
shell: the result moves to a higher layer before it can close
escrow: the result is preserved until a later dependency is available
delay: the result is valid only after a time/window condition is satisfied
boundary collision: the result marks where the current frame stops
```

These categories keep the reader from confusing an unfinished route with a
discarded result.

## Conclusion

Paper 0.75 defines how the first toolkit and claim contract are applied to the
next state. The rule is simple: no connection is hidden. If a later paper uses
an earlier result, it must say how that result is being carried.
