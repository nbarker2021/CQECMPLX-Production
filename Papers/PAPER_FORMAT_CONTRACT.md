# CQECMPLX Paper Format Contract

## Purpose

This contract defines the review-facing structure for the CQECMPLX papers.
The papers must read as polished formal scientific papers. Tooling, kernels,
analog workbooks, and narrative aids are supporting evidence, not the main
argument.

## Paper 0

Paper 0 is a foreword. It states the authorial position plainly:

- Nicholas "Nick" Barker is self-trained.
- AI systems were used extensively to assist the work.
- The work builds from trained academic and scientific sources that deserve
  primary respect as the discovered foundations.
- Extraordinary claims require extraordinary proof.
- The package is built with unusually heavy receipts, analog reproducibility,
  executable tests, and open obligations because the author is not asking for
  trust by status.

Paper 0 is not the proof of the system. It explains why the proof burden is
handled the way it is.

## Quarter Papers

Each section may include `.25`, `.50`, and `.75` companion papers.

`N.25` is the toolkit paper for the upcoming section. It describes what a
reader can use: software tools, analog kit, kernels, lib bindings, receipts,
and expected inputs/outputs.

`N.50` is the claim contract paper. It defines what must be true before a
claim is treated as a claim rather than a boundary failure, candidate, or
obligation. It also provides the Merkle/link review layer between neighboring
papers.

`N.75` is the application/precondition paper. It explains how the tools and
results from the previous interval become preconditioned settings for the next
paper: higher shell, escrow, delay, boundary collision, or next-state framing.

Quarter papers may speak more directly to a reader. Integer papers should not
lean on that voice.

## Integer Papers

Each integer paper after Paper 0 is a scientific paper. It must prioritize:

1. Abstract.
2. Main claims.
3. Predictions.
4. Test protocol.
5. Generated proof or receipt.
6. Formalized claim.
7. Falsifiers and scope boundaries.
8. Results.
9. References and evidence bindings.

Each paper may expose support systems, but the support systems should remain
secondary unless the paper's claim is itself about a tool.

## The Observer Event

All papers preserve the same core explanation:

```text
an observer chooses an enumeration event
-> the event defines C
-> a state is recorded
-> the record is validated
-> the result is interpreted
```

Nothing after the initial observer-chosen enumeration event may be smuggled in
as an unexplained addition. Claims must port across the paper sequence when
used, and they must have an analog or by-hand exposure route where possible.
That requirement is not meant to slow the reader down. It exists to show that
the work is built from basic topological reading, indexing, and receipt
preservation rather than from hidden machinery.

## Archive Rule

Nothing in `D:\` is discarded until it is inventoried and its value is
extracted. Cleanup means:

```text
inventory -> classify -> extract -> promote/archive -> receipt
```

It does not mean deleting unreviewed material.
